# YBC ERP — Полное ТЗ + Оркестрация + Статус выполнения

**Версия:** 1.0 (живой документ)
**Дата обновления:** 2026-03-01
**Источник:** текущий рабочий контур проекта + фактически сделанные коммиты

---

## 1) Цель проекта

Собрать production-grade ERP для YBC (Cyprus) с управленческим учетом и бухгалтерским контуром:
- Memberships
- Events & Tickets
- Bills & Payments (accrual)
- GL (двойная запись)
- Кассовые документы (ПКО/РКО)
- Обмен валют
- Перемещения между кассами/кошельками
- Sandbox + demo data + reset

UI: **строгий Apple/macOS стиль**, украинский язык, без «игрушечных» элементов.

---

## 2) Архитектурные решения (зафиксировано)

## 2.1 Технический стек
- Frontend: React + Vite
- Data/Auth: Supabase (Postgres + Auth + RLS)
- Hosting: Cloudflare Pages (домен `erp.ybc.com.cy`)
- Railway: **не используется в целевой архитектуре**

## 2.2 Принципы
- Tenant isolation через `tenant_id`
- RLS на таблицах
- Все операции учета должны вести к проводкам GL
- Все документы в едином журнале документов

---

## 3) Оркестрация разработки

```text
[Orchestrator]
   ├─ Plan (разбивка на этапы, acceptance criteria)
   ├─ Build (Backend/Data + Frontend)
   ├─ QA (проверка сценариев и регресс)
   ├─ Fix (исправление дефектов)
   └─ Sign-off (готовность этапа)
```

Цикл:
```text
PLAN -> BUILD -> QA -> FIX -> (повтор) -> DONE
```

---

## 4) Блок-схемы (функциональные)

## 4.1 Auth Flow
```text
User -> /login -> Supabase Auth signInWithPassword
     -> session OK
     -> if force_password_change=true -> /change-password
     -> update password -> /dashboard
```

## 4.2 Membership Plan Creation
```text
UI Modal -> validate -> Supabase insert membership_plans
        -> reload list -> visible in grid
```

## 4.3 Membership Lifecycle
```text
Create membership
   -> start_date/end_date/payment_amount
   -> freeze (optional)
      -> insert membership_freeze
      -> extend end_date
   -> cancel (status=cancelled)
```

## 4.4 GL Posting (целевая модель)
```text
Document (Bill/PKO/RKO/Exchange/Transfer)
   -> generate balanced journal entries (Dt/Kt)
   -> write document_journal + journal_entries
   -> enforce debit==credit
```

## 9) Definition of Done (глобально)

Система считается «готовой», когда:
- все 6 шагов в статусе [x]
- нет пустых экранов и критических ошибок сохранения
- все финансовые операции проходят через GL
- sandbox reset работает
- QA pass по ключевым сценариям
- mobile-first требования из раздела 8 закрыты

## 10) Приложение: рабочий режим оркестра

```text
Оркестратор = single source of truth
  - фиксирует план
  - координирует реализацию
  - не закрывает этап без QA
  - ведет публичный прогресс (что сделано/проверено/дальше)
```

Этот документ — контрольная точка: показывает, что понимаем, что делаем, где готовы, где нет.
