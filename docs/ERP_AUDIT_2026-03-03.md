# YBC ERP — End-to-End Audit (2026-03-03)

## 1) Executive Summary
- Загальний стан: **працює, але QA-процес не зафіксований технічно**.
- Продакшен доступний: `https://erp.ybc.com.cy`
- Ризики: крос-браузерні CSS-регресії, відсутність CI QA gate, ручний деплой.

## 2) Реалізовано
1. 12+ функціональних модулів (Dashboard, Memberships, Events, Bills, Wallets, Docs, COA, Reports, Users, Settings).
2. Supabase схема + RLS + GL auto-posting trigger.
3. Публічна реєстрація, інвайти, reset-password flow.
4. Blockchain інтеграція (EVM/L2/Tron/Bitcoin) + баланс/tx.
5. Mobile-first UI з finder-підходом.

## 3) Частково / потребує добивки
1. Формальний E2E QA pipeline (Chromium+Firefox) — відсутній.
2. Автоматичний деплой через GitHub Actions — workflow disabled.
3. Регресійні автотести в репозиторії (jest/playwright) — відсутні в runtime.
4. Signup flow: знайдений дефект узгодження auth.users vs public.users (залежить від схеми constraints).

## 4) Дефекти, підтверджені аудитом
- Firefox стилі: `prefers-color-scheme: dark` викликав білий/світлий текст у світлій темі.
- Mobile membership form: переповнення date/input блоку на iOS Safari.
- Signup: початковий сценарій читав tenants із публічного клієнта -> RLS помилка.

## 5) Що вже виправлено в рамках аудиту
- Переведено темні стилі проблемних сторінок на `:root[data-theme='dark']`.
- Підправлено mobile header/button align та back color (#FA5255).
- Усунуто mobile overflow у Membership form.
- Signup: прибрано залежність від читання tenants у публічному потоці.
- Додано в UI модуль **"Довідка ERP"** (`/manual`) з табами:
  - Огляд системи
  - Модулі та функції
  - Бізнес-логіка
  - Онбординг ролей
  - FAQ

## 6) Наступні кроки (in progress)
1. Прогін тестових даних по всіх модулях (seed + smoke flows).
2. Серія скриншотів по кожному модулю (desktop + mobile).
3. Повна user guide (in-app + docs markdown).
4. Рольовий онбординг (Owner/Admin/Staff/Accountant/Event Manager/Cashier/Analyst).
5. Підключення QA gate в CI.

## 7) Оркестрація: чому QA пропускав
- Процес описаний документально, але не enforced у pipeline.
- Відсутній hard gate `QA PASS` перед deploy.
- Ручний деплой дозволяв випуск без повного крос-браузер smoke.
